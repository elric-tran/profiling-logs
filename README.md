# ProfilingLogs

MiniProfiler-based SQL profiling for ASP.NET Core + EF Core, with developer-friendly extras.

## Why ProfilingLogs?

- **A tiny live widget on every page** — shows the execution time of the UI, the API and,
  most importantly, the **SQL statements generated by EF Core** in real time.
- **Tied to your application's context** — each SQL statement is linked directly to the HTTP
  request or the specific UI action that triggered it. You know *exactly* which SQL EF Core emits
  when you click the "View details" button.
- **Catch N+1 queries fast** — a single action that fires dozens of near-identical queries is
  immediately obvious in the timing tree.
- **See the real parameters** — inspect the actual parameter values sent to the database, not just
  `@__p_0` placeholders.
- **Jump to the exact line of code** — every query is annotated with `File (Line) -> Method` and a
  clickable IDE deep-link that opens the source file at that line (VSCode / Cursor / Rider / custom).
- **Non-SQL steps too** — wrap and time arbitrary code blocks, HTTP calls, cache lookups, etc.
- **Per-connection color markers** — every connection Open/Close pair gets a stable color emoji
  (reused pooled connections get a fresh color each open).
- **Extremely lightweight & easy to configure** — two lines in `Program.cs`.
- **Production-safe toggle** — everything is a no-op unless `Enabled` is set, so it can ship to
  Production with zero overhead and no source-path leakage.

## What you see in the logs

For every query, ProfilingLogs shows the **generated SQL** *and* the exact line of code that
executed it — and that line is a **clickable link** that opens the file directly in your IDE.

Say you have a service like this:

```csharp
// Example.Service.cs
public class ExampleService
{
    private readonly AppDbContext _db;

    public ExampleService(AppDbContext db) => _db = db;

    public async Task<List<User>> GetActiveUsersAsync()
    {
        return await _db.Users                       // Line 27
            .Where(p => p.IsActive)
            .ToListAsync();
    }
}
```

The captured EF Core query is annotated with the caller's `File (Line) -> Method` and an IDE
deep-link, so the profiler output looks like:

```sql
-- 🔗 From: Example.Service.cs (Line 27) -> GetActiveUsersAsync
-- 🔗 vscode://file/C:/src/MyApp/Services/Example.Service.cs:27
SELECT [p].[Id], [p].[IsActive]
FROM [Users] AS [p]
WHERE [p].[IsActive] = CAST(1 AS bit);
```

On the results page (`{RouteBasePath}/results-index`) the `vscode://file/...:27` link is rendered
as a clickable hyperlink — **clicking it opens `Example.Service.cs` at line 27 directly in your
IDE** (VSCode / Cursor / Rider, per the `Ide` option). No more hunting for which method produced
a slow query.

## Install

```bash
dotnet add package ProfilingLogs
```

## Configuration

```csharp
using ProfilingLogs;

// Program.cs
builder.Services.AddProfilingLogs(o =>
{
    o.Enabled = builder.Environment.IsDevelopment(); // flag from outside
    o.Ide     = ProfilingIde.VSCode;                 // default
});

var app = builder.Build();

app.UseProfilingLogs();
```

Or bind from configuration:

```csharp
builder.Services.AddProfilingLogs(builder.Configuration.GetSection("ProfilingLogs"));
```

```json
{
  "ProfilingLogs": {
    "Enabled": true,
    "RouteBasePath": "/profiler",
    "Ide": "Cursor"
  }
}
```

View results at `{RouteBasePath}/results-index` (default `/profiler/results-index`).

## Options

| Option | Default | Description |
|---|---|---|
| `Enabled` | `false` | Master switch. Pass `IsDevelopment()`. When false everything is a no-op. |
| `RouteBasePath` | `/profiler` | MiniProfiler base path. |
| `EnableVsCodeLinks` | `true` | Linkify IDE deep-links in the results page. |
| `EnableConnectionColors` | `true` | Color markers for connection Open/Close. |
| `EnableCallerComment` | `true` | Inject `-- 🔗 From ...` comment into SQL. |
| `HideDefaultConnRows` | `true` | Hide default `sql - Open/Close` rows. |
| `EnableClearCacheButton` | `true` | Show a "Clear all profiler results" button on the results-index page. |
| `SqlNamespaceFilter` | `"Services"` | Only walk stack frames whose type namespace contains this. |
| `Ide` | `VSCode` | `VSCode` / `Cursor` / `Rider` / `VisualStudio` / `Custom`. |
| `IdeUrlFormat` | `null` | Custom template, overrides preset. Placeholders `{path}` `{line}` `{col}`. |
| `PathMap` | `null` | Map remote→local path prefixes (viewing from another machine). |
| `ColorScheme` | `Dark` | MiniProfiler color scheme. |

## Notes / Caveats

- `EnableCallerComment` appends a comment to the **executed** SQL text. This is valid T-SQL but changes the command text per call-site, which can reduce query-plan reuse. Keep it gated behind `Enabled = IsDevelopment()`.
- IDE deep-links only resolve correctly when the browser runs on the **same machine** as the source (or `PathMap` is configured). Paths containing spaces are URL-encoded automatically.
- `ProfilingIde.VisualStudio` uses a `devenv://` link. Visual Studio has no native URL scheme, so it requires a one-time protocol-handler registration — see [`tools/README.md`](tools/README.md). `VSCode` / `Cursor` / `Rider` work out of the box.
- The results-index page shows a **Clear all profiler results** button (top-right) that wipes every captured request. Disable it with `EnableClearCacheButton = false`.
- GitHub Packages requires authentication (PAT) even for public packages when restoring.

## Target frameworks

`net8.0`, `net9.0`, `net10.0`.
